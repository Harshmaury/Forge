## Capability Domain

Forge is the execution domain of the developer platform.

Platform-wide capability boundaries:
  ~/workspace/architecture/platform-capability-boundaries.md

Platform ADRs that govern Forge:
  ADR-001 — Nexus is the canonical project registry
  ADR-003 — HTTP/JSON on 127.0.0.1:8082
  ADR-004 — Command model is the core abstraction

---

## What Forge Owns

- Command intake, validation, and translation
- Intent execution pipeline
- Workflow definition storage and execution (Phase 2)
- Event-driven automation triggers (Phase 3)

## What Forge Does Not Own

- Runtime service state (Nexus)
- Service start/stop authority (Nexus)
- Project registry (Nexus — ADR-001)
- Filesystem observation (Nexus — ADR-002)
- Workspace knowledge and indexing (Atlas)
- Architecture conflict detection (Atlas)

Forge never starts or stops services directly.
It instructs Nexus via the Nexus HTTP API.

## Technology Stack

Language:    Go 1.23.0
HTTP server: net/http stdlib
UUID:        github.com/google/uuid — command IDs (ADR-004)
SQLite:      mattn/go-sqlite3 — workflow storage (Phase 2)
YAML:        gopkg.in/yaml.v3 — workflow definitions (Phase 2)
CLI:         github.com/spf13/cobra — same as Nexus engx
Nexus:       github.com/Harshmaury/Nexus/internal/eventbus
             topic constants only, for Phase 3 automation triggers

### Why Go

Consistency with Nexus and Atlas. Same build toolchain, same HTTP
pattern, same deployment model. os/exec for command execution is
natural in Go.

### Why uuid

Every command object requires a stable id (ADR-004). The uuid package
is the lightest correct solution. Command IDs enable tracing, idempotency
checks, and Phase 2 workflow step references.

### Why cobra (Phase 1 only as forge CLI)

Forge exposes its own CLI binary (forge) for direct invocation.
engx delegates to Forge via HTTP. The forge binary uses cobra
consistently with how engx works in Nexus.

## Intent Model (ADR-004)

### Command Object Schema

Every command — from CLI, HTTP API, or automation trigger — is
translated into this structure before the execution engine receives it:

```json
{
  "id":         "<uuid>",
  "intent":     "<action name>",
  "target":     "<project or service id>",
  "parameters": { "<key>": "<value>" },
  "context":    {
    "workspace_root":   "/home/harsh/workspace",
    "project_path":     "/home/harsh/workspace/projects/apps/nexus",
    "language":         "go",
    "requesting_agent": "cli",
    "timestamp":        "2026-03-15T04:23:00Z"
  }
}
```

Field rules:
- id: generated by Forge if not supplied — never nil
- intent: must match a registered intent handler
- target: validated against Nexus project registry
- parameters: intent-specific, validated per handler
- context: populated from Atlas + Nexus if not supplied by caller

### Translation Layer

The translation layer is the boundary between Forge's interface
and its execution engine. All entry points pass through it:

```
CLI command     → translator → Command object → executor
HTTP POST       → translator → Command object → executor
Event trigger   → translator → Command object → executor (Phase 3)
```

The executor never receives raw strings.

## Implementation Phases

### Phase 1 — Command Execution

Single commands, immediate execution, no persistence.

Supported intents:
  build   run build command for target project language
  test    run test suite for target project
  run     start a process (via Nexus process provider)
  deploy  coordinate deployment sequence via Nexus

Execution pipeline:
  1. Receive raw input (CLI flag or HTTP body)
  2. Translate to Command object (validate all fields)
  3. Resolve target (GET http://127.0.0.1:8080/projects/:id)
  4. Enrich context (GET http://127.0.0.1:8081/workspace/project/:id)
  5. Execute intent handler
  6. Return structured result

Phase 1 is stateless — no command history stored.

### Phase 2 — Workflow Definitions

Named sequences of commands, stored in SQLite.

Workflow schema:
```json
{
  "id":      "<uuid>",
  "name":    "full-deploy",
  "steps":   [<command>, <command>],
  "trigger": "manual"
}
```

Phase 1 command schema is unchanged — workflows wrap commands.
No breaking schema change (by design — ADR-004).

### Phase 3 — Automation Triggers

Event-driven. Forge subscribes to Nexus event bus topics.
Maps workspace events to workflow definitions.

```json
{
  "event":    "workspace.file.modified",
  "filter":   { "extension": ".go" },
  "workflow": "run-tests"
}
```

## Interaction with Nexus

- Resolve project state: GET http://127.0.0.1:8080/projects/:id
- Request service actions: POST http://127.0.0.1:8080/projects/:id/start
- Forge does not import Nexus internal packages (except eventbus in Phase 3)
- Forge does not write to the Nexus SQLite database

## Interaction with Atlas

- Enrich command context: GET http://127.0.0.1:8081/workspace/project/:id
- Forge reads from Atlas — never modifies Atlas indexes

## HTTP API (127.0.0.1:8082)

Response envelope: { ok, data, error }

Phase 1:
  POST /commands              submit command, returns execution result
  GET  /commands/:id          retrieve result by command ID
  GET  /intents               list registered intent handlers
  GET  /health                liveness probe

Phase 2:
  POST /workflows             create workflow definition
  GET  /workflows             list stored workflows
  GET  /workflows/:id         retrieve workflow
  POST /workflows/:id/run     execute workflow

Phase 3:
  POST /triggers              register event-to-workflow mapping
  GET  /triggers              list active triggers
  DELETE /triggers/:id        remove trigger

## Forge Design Principles

1. Forge acts — executes intent, Atlas reads, Nexus coordinates
2. Forge translates — all input becomes a Command object before execution
3. Forge delegates — service lifecycle decisions go through Nexus
4. Forge phases — workflow and automation layers build on command layer
5. Forge is stateless in Phase 1 — no persistence until Phase 2 begins
